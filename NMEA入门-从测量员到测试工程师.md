# NMEA 入门:从测量员视角到测试工程师视角

> 这份文档把我们这次对话从头到尾梳理成一条线,方便你反复看。
> 核心线索:**你以前用 RTK 是站在数据管道的顶端(用成品),现在转测试岗是要钻进管道中间(测原料和加工过程)。NMEA 就是那个"原料"。**

---

## 一、NMEA 是什么

NMEA-0183 是美国国家海洋电子协会定的一套**文本协议**。GNSS 接收机(包括中海达的 RTK 主机)默认就用它把定位结果往外吐。

特点:

- 纯文本,一行行的,每行叫一条**语句(sentence)**
- 每条以 `$` 开头,紧跟 `讲话人 + 类型`(如 `GPGGA`),字段用逗号隔开,末尾 `*` 加**校验和**
- 通过串口 / 蓝牙 / TCP 发出去

常见的几种语句:

| 语句 | 装什么 | 有坐标吗 |
|------|--------|----------|
| `GGA` | 定位核心:经纬度、海拔、卫星数、定位质量 | ✅ 有 |
| `RMC` | 推荐最小定位:经纬度、速度、航向、UTC 时间 | ✅ 有 |
| `GSA` | 这次用了哪些卫星 + 精度因子(DOP) | ❌ 无 |
| `GSV` | 每颗可见卫星的详情(仰角、方位角、信噪比) | ❌ 无 |

**关键认知:同一秒里只有 GGA 和 RMC 带坐标,而且它俩描述同一个位置、值一致。其余语句没有坐标列。**

---

## 二、一条 GGA 怎么拆

标准示例:

```
$GPGGA,123519,4807.038,N,01131.000,E,1,08,0.9,545.4,M,46.9,M,,*47
```

逗号分隔,逐字段:

| 字段 | 含义 |
|------|------|
| `$GPGGA` | 语句头:GGA = 定位基本信息 |
| `123519` | UTC 时间 12:35:19 |
| `4807.038,N` | 纬度 48°07.038′ 北 |
| `01131.000,E` | 经度 11°31.000′ 东 |
| `1` | **定位质量**(RTK 重点看这个):0=无 1=单点 4=RTK固定 5=RTK浮动 |
| `08` | 参与定位的卫星数 = 8 颗 |
| `0.9` | HDOP 水平精度因子(越小越好) |
| `545.4,M` | 海拔高度 545.4 米 |
| `*47` | 校验和(检验数据是否传错) |

> **解析的本质 = 切字符串 + 按格式查表取值。** 难度不高,但踩在你的领域知识上。

## 三、NMEA 在 RTK 作业里的位置

你以前没印象 NMEA,是因为它被**手簿软件(Hi-Survey 那类)藏在底下了**。真实数据流:

```
卫星 + 基站(信号与差分改正)
        ↓
RTK 接收机(主机)  ← NMEA 就在这层产生
        ↓  蓝牙 / 串口传 NMEA
手簿软件(Hi-Survey):解析 NMEA + 坐标转换
        ↓
你看到的坐标点(按"采集"存点)  ← 你以前的视角在这
        ↓
导出 .dxf / .dat → 进 CAD 编辑
```

两个要点:

1. **你在 CAD 里看到的坐标,已经不是 NMEA 原始值了。** GGA 给的是 WGS84 经纬度;CAD 要的是当地平面坐标。中间那步**坐标转换(七参数/四参数)**是手簿干的——这正好是你熟的东西。
2. **转测试岗 = 身份从管道顶端掉到管道中间。** 以前你是用软件的人,现在你要测的就是手簿/主机这层软件:验证它解析对不对、坐标转换准不准、信号异常处理得对不对。**你现在要看的,正是当年看不到的那一层。**

---

## 四、NMEA 是持续流,不是按键才发

**重要纠错:NMEA 不是你按采集才发的,它一直在发。**

- 只要主机开着、连着手簿,它就**持续不断**地吐 NMEA,通常每秒 1 条(1Hz),好的接收机能到 5Hz / 10Hz / 20Hz
- 你发呆、走路、抽烟,它照样每秒吐
- 你按"采集"按钮,**不是让它开始发**,而是让软件**抓住那一瞬间正流过来的那一组**,存成一个点

这就是为什么日志"数量众多"——持续每秒发 × 每个历元好几条语句,日志堆得飞快。

---

## 五、一个"历元"里有哪些语句

**历元(epoch)= 同一秒、同一时间戳的一组语句。**

同一历元主机会吐好几种语句,但:

- 真正带坐标的只有 GGA(和 RMC),它俩值一致
- GSA / GSV 不带坐标,管的是卫星和精度

你看 GGA 和 RMC 里的时间戳都是 `123519`,就是靠这个**同一时间戳**把它们绑成一个历元。

---

## 六、四种语句各自的格式(不一样!)

它们唯一相同的是"外壳"(`$` + 类型 + 逗号字段 + `*校验和`),**逗号中间装的东西每种完全不同**:

```
$GPGGA,123519,4807.038,N,01131.000,E,1,08,0.9,545.4,M,46.9,M,,*47
$GPRMC,123519,A,4807.038,N,01131.000,E,022.4,084.4,230394,003.1,W*6A
$GPGSA,A,3,04,05,,09,12,,,24,,,,,2.5,1.3,2.1*39
$GPGSV,2,1,08,01,40,083,46,02,17,308,41,12,07,344,39,14,22,228,45*75
```

- **GGA**:时间、纬度、经度、定位质量、卫星数、HDOP、海拔……
- **RMC**:时间、`A`(有效)、纬度、经度、`022.4`(速度/节)、`084.4`(航向)、`230394`(日期)、磁偏角。也有经纬度,但后面排列跟 GGA 完全不同
- **GSA**:`A`(自动)、`3`(3D定位)、一串卫星编号、最后 PDOP/HDOP/VDOP
- **GSV**:`2`(共2条)、`1`(第1条)、`08`(可见8颗),后面**每4个一组**描述一颗星(编号、仰角、方位角、信噪比)

> **解析第一步永远是:先看头(`$GPxxx`)判断类型,再用对应规则切字段。** GGA 第2字段是纬度,RMC 第2字段却是状态位,拿错规则全错。

**讲话人标识(开头两个字母):** `GP`=GPS、`BD`/`GB`=北斗、`GL`=GLONASS、`GA`=伽利略、`GN`=多系统合并解。中海达多星座设备,你以后大概率满屏 `BD` 和 `GN`。

---

## 七、语句的发送顺序与"历元"

**它们不是同时蹦出来的,是一条接一条、按固定顺序串行发的。**

蓝牙/串口是"单车道",字符只能排队过。所以一秒里:先发完 GGA → 再 GSA → 再 GSV → 再 RMC……走完,下一秒整组重复。

- **有先后**:顺序由设备固件决定,同一台设备通常固定
- **同一历元**:靠同一时间戳绑在一起
- "传输顺序"和"历元归属"是两回事,不矛盾

---

## 八、采集时到底记录了什么

按采集键 = 锚定它所属的那个历元,用这一组语句生成一个点。但要拧精确:

1. **记录的不是那几条原文,而是解析、算出来的结果**——主要是坐标,外加质量标签(定位质量、卫星数、HDOP)。原始 NMEA 一般另存为单独日志。语句是"原料",点记录里是"成品"。
2. **坐标通常是坐标转换后的当地平面坐标**(给 CAD 用),不是 GGA 的原始 WGS84。
3. **常常跨多个历元取平均**(平滑采集),让点更稳——这是优化细节,不影响主干。

一句话:**采集 = 锁定所属历元 → 解析出坐标 → 坐标转换 → 连同质量标签存成一个点。**

---

## 九、为什么语句数量会变多(不是重复!)

一个历元正常**不会出现两三条 GGA,GGA 永远一条**。多出来的语句来自两种正经机制:

1. **GSV 续号**:一条 GSV 最多装 4 颗星。可见 11 颗就拆成 1/3、2/3、3/3 三条,每条装不同的星——是"接续"不是"重复"。
2. **多星座分系统**:GPS、北斗、GLONASS 各算各的,每个系统各报一份 GSA + GSV(`$GP..`、`$BD..`),位置则合并成一条 `$GN..`。

真实多星座历元示意(校验和用 `*..` 占位):

```
$GNGGA,123519,4807.038,N,01131.000,E,4,18,0.6,545.4,M,...*..   ← 位置,就 1 条
$GPGSA,A,3,04,05,09,12,...,1.8,0.6,1.2*..    ← GPS 用的星
$BDGSA,A,3,07,10,13,...,1.8,0.6,1.2*..       ← 北斗用的星(不同系统,非重复)
$GPGSV,3,1,11,...*..    ← GPS 可见星 第1条
$GPGSV,3,2,11,...*..    ← 第2条
$GPGSV,3,3,11,...*..    ← 第3条
$BDGSV,2,1,07,...*..    ← 北斗可见星 第1条
$BDGSV,2,2,07,...*..    ← 第2条
$GNRMC,123519,A,4807.038,N,01131.000,E,...*..  ← 1 条
```

9 行,但**位置信息(GNGGA)始终只有 1 条**,没有任何一条是另一条的复制品。

## 十、测试 NMEA:两种测试 + 两种真值

这是从"测量员思维"切到"测试工程师思维"最关键的一道坎。

### 两种测试

| | 测软件解析对不对 | 测设备精度准不准 |
|---|---|---|
| 测什么 | 软件有没有把字段切对、异常处理对不对 | 解算出的坐标准不准 |
| 真值来源 | **你自己按格式算的期望值** | **控制点(地上钉子)的实测坐标** |
| 需要钉子吗 | ❌ 不需要 | ✅ 需要 |
| 谁的活 | **你这个软件测试岗的主战场** | 偏测绘/质检验收 |

### 关键:测解析逻辑,不需要真实坐标

这时候的"真值"不是从卫星或钉子来的,是**你自己按 NMEA 标准算的**。

例:手写一条纬度字段 `4807.038,N`,按格式(度度分分.分分分)它就**必须**解析成
`48 + 07.038 ÷ 60 = 48.1173°`。
这个期望值跟现实世界那个点在哪儿毫无关系。你测的是:**"给程序这串文本,它读对了没有。"**

### 为什么"自己造数据"比"用真实日志"更好

你能造现实中几个月都遇不到一次的极端情况:

- 纬度写成 95°(超范围,该报错)
- 校验和故意写错(该被拦下)
- 句子缺字段 / 半截断掉
- 定位质量从 4 突然掉到 1(信号异常)
- 同一历元混进重复 GGA(数据错乱)

> **真值不是从天上(卫星)来的,是你自己构造的。** 你做过测量、懂坐标转换,造这些用例时对"什么合理、什么是坑"特别有谱——这是你的差异化优势。

### 什么时候才用钉子坐标

当你想验**整条链路精度**时:拿一段真在钉子上录的日志,把钉子已知坐标当期望值,测"解析 + 坐标转换"出来的点差多少。这叫端到端精度测试,通常是验收环节,不是日常单元测试的主菜。

## 十一、测试要盯的坑(汇总)

把全文里提到的测试点集中一下,这些就是写测试用例时要覆盖的:

1. **字段解析正确性**:每个字段按对应语句的格式切对、值对。
2. **类型识别**:先认 `$GPxxx` 头,用对规则。拿 GGA 规则去解析 RMC 会全错。
3. **校验和**:`*` 后的校验和对不上 → 数据传输出错,该被识别。
4. **数值范围**:纬度 ≤90、经度 ≤180、定位质量在合法集合内等。
5. **历元归属 / 错配**:必须按时间戳把多行正确归到同一历元。经典 bug:第 N 秒的 GGA 配第 N+1 秒的 GSV。
6. **语句缺失**:某秒 GGA 没发,会不会拿上一秒旧位置硬配这一秒卫星状态?
7. **半截数据 / 截断**:`$GPGGA,123519,4807.0` 戛然而止,解析器崩不崩?
8. **合法的多条 vs 非法的重复**:
   - 多条 GSA / GSV(分系统、续号)是正常的 → 要正确归并,别当错误
   - 同一历元出现重复 GGA 是异常 → 要能抓出来并报警
   - 注意"同一历元"指**同一精确时刻**（含小数秒），不是指同一整秒。5Hz 接收机 083000.00 和 083000.20 是两次各自独立的定位，不应判为重复
9. **定位质量跳变**:从 RTK 固定(4)掉到单点(1)——RTK 设备测试的重中之重。

## 附:关键术语速查

- **NMEA-0183**:GNSS 设备输出定位数据的文本协议
- **语句 / sentence**:一行 NMEA 数据,`$` 开头 `*校验和` 结尾
- **历元 / epoch**:同一时间戳的一组语句,代表某一瞬间的一次定位
- **讲话人标识 / talker ID**:头部两字母,标系统(GP/BD/GL/GA/GN)
- **GGA**:含坐标的核心定位语句
- **定位质量**:GGA 里那位,0无/1单点/4RTK固定/5RTK浮动
- **HDOP / PDOP / VDOP**:精度因子,越小越好
- **解析(parse)**:把文本切成结构化字段
- **校验(validate)**:用规则判断字段合不合理
- **真值**:测解析 = 自己算的期望值;测精度 = 控制点实测坐标

---

*你的优势一句话:做过 RTK 测量、懂坐标系投影和坐标转换,对每个字段该长啥样心里有数。这套领域知识,是纯计算机背景的测试工程师不具备的——把它用在 NMEA 解析/校验的测试上,做起来又快又稳。*
